Data-driven runs

pgwen runs the same scenario once per record from a CSV/JSON feed or an Examples table. Three mechanisms cover most needs.

Examples tables #

Inline Gherkin Examples: blocks under a Scenario Outline. Best for a small, fixed set of variants.

Scenario Outline: Adding "<ITEM>" shows count "<COUNT>"
  Given I navigate to "https://todomvc.com/examples/react/"
  When  I enter "<ITEM>" in the new-todo input
  And   I press the Enter key
  Then  the todo count should contain "<COUNT>"

  Examples:
    | ITEM        | COUNT |
    | First task    | 1     |
    | Second task   | 1     |
    | Walk dog    | 1     |

CSV feed (-i) #

Drive scenarios from an external CSV file. Each row becomes one scenario execution; column headers become scope bindings.

# pgwen/input/todos.csv
ITEM,EXPECTED_COUNT
First task,1
Second task,1
Walk dog,1
yarn pgwen -i pgwen/input/todos.csv pgwen/features/todo.feature

Inside the scenario, reference columns by binding name: ${ITEM}, ${EXPECTED_COUNT}.

JSON feed #

Same idea but with a JSON array. Each object's keys become scope bindings.

[
  { "ITEM": "First task",   "EXPECTED_COUNT": "1" },
  { "ITEM": "Second task",  "EXPECTED_COUNT": "1" }
]
yarn pgwen -i pgwen/input/todos.json pgwen/features/todo.feature

@Examples('<path>') #

Attach a feed to a single StepDef (not the whole feature). Optional where= predicate filters rows:

@StepDef
@Examples('${pgwen.baseDir}/input/orders.csv', where="REGION=='NSW' && STATUS=='ACTIVE'")
Scenario Outline: I process order "<ORDER_ID>"
  Given I navigate to "${app.url}/orders/<ORDER_ID>"
  Then  the result panel should be displayed

autoTrim #

Surrounding whitespace in CSV/JSON values is auto-trimmed by default for CSVs. Configurable per format:

pgwen.auto.trim.data {
  csv  = true     # default
  json = false    # default — preserve whitespace
}

Iteration context #

Inside a data-driven scenario, the implicit bindings expose iteration state:

BindingValue
pgwen.data.record.index0-based index of the current record.
pgwen.data.record.number1-based number of the current record.
pgwen.iteration.index / .numberSame, for @ForEach bodies.
pgwen.examples.table.record.index / .numberSame, for Examples: tables under a Scenario Outline.

Read-only feed scope #

pgwen.input.data.readOnly = true (default) prevents accidental reassignment of feed-derived bindings inside the scenario. Useful for catching name collisions early.

Per-record reporting #

The HTML report renders one row per record under the scenario, with status, duration, and any failure attachment.

Worked example — CSV feed + Scenario Outline #

# pgwen/input/customers.csv
customer_id,plan,expected_status
C-001,basic,active
C-002,premium,active
C-003,trial,pending
# pgwen/features/customer-status.feature
@Examples(file='pgwen/input/customers.csv')
Feature: Customer status

  Scenario Outline: account <customer_id> on the <plan> plan is <expected_status>
    Given I am on the admin dashboard
     When I search for account "<customer_id>"
     Then the plan badge should be "<plan>"
      And the status indicator should be "<expected_status>"

Each CSV row becomes a separate scenario iteration. The HTML report groups them under the outline with one collapsible row per record; the failure-class badge + sibling-success-rate chip apply per record so a row that fails because of bad input data is visually distinct from a row that fails because of a real bug. Filter rows mid-feed with @Examples(file='…', where='<expr>'): 

@Examples(file='pgwen/input/customers.csv',where="plan eq 'premium'")
Feature: Premium-only checks
