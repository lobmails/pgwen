pgwen diagnose

Every failed step is automatically tagged with a rule-based failure class (LOCATOR_NOT_FOUND, ASSERTION_FAILED, TIMEOUT, AUTH_FAILURE, NAVIGATION_FAILURE, UNKNOWN) — visible as a coloured badge in the HTML report's error panel and as failureClass in results.json. The pgwen diagnose CLI builds on that signal to give the on-call dev something concrete to read at 2am.

Four modes #

The mode is decided by the flag you pass (or by absence of a flag).

FlagWhat it doesReadsNetwork?
--rules-onlyPer-failure rule-based reportresults.jsonNo
(none)Per-failure Claude-backed reportresults.json + pgwen.confYes (gated)
--reportAggregate telemetry analysisdiagnosis-history/ + diagnosis-cache/No
--annotateInteractive review of historydiagnosis-history/No

AI mode falls back to --rules-only output with a notice whenever any gate fails — disabled in config, kill-switch env set, no API key, no pgwen.conf in the working dir. The CLI never produces an empty report and never silently calls Claude.

AI mode groups failures by pattern before calling Claude — a single broken locator that fired in 500 scenarios costs one Claude call, not 500. The diagnosis is fanned out to every member of the group, and the per-entry instances list carries the full blast radius downstream to pgwen-fix. See Pattern-level grouping below.

Rules-only mode #

Free, offline, no setup. Useful as a smoke check after any run that produced failures. 

$ pgwen diagnose --rules-only --input pgwen/output/results.json

AI-assisted mode #

Opt in by setting pgwen.diagnose.ai.enabled = true in pgwen.conf and exposing an Anthropic API key. Then run pgwen diagnose with no flag.

Minimum setup #

The project's pgwen.conf needs at least the enable flag:

pgwen.diagnose.ai {
  enabled = true
}

The API key comes from one of three places (highest precedence first):

  1. An environment variable (default name ANTHROPIC_API_KEY; rename via pgwen.diagnose.ai.api.keyEnv).
  2. A pgwen.conf entry, required to use the :masked suffix so the value is redacted in logs.
  3. A programmatic override (used by tests and external secret stores).

Recommended layout — key in env, never on disk:

# in your shell or CI secret
export ANTHROPIC_API_KEY="sk-ant-..."

# in pgwen.conf
pgwen.diagnose.ai {
  enabled = true
}

If the env path isn't available, the config fallback:

pgwen.diagnose.ai {
  enabled = true
  api {
    "key:masked" = "sk-ant-..."   # `:masked` is REQUIRED — pgwen warns otherwise
  }
}
What :masked protects (and what it doesn't). The suffix flags a value as sensitive: pgwen replaces it with ***** wherever the StringInterpolator renders for display (logs, HTML report, console output) and pgwen diagnose reports the key as a fingerprint like config:sk-...3f2a, never the full value. However, the value is still stored verbatim in pgwen.conf — :masked is not encryption. Anyone with read access to the file reads the key. Source-control commits expose it forever. For any production secret, use the env-var path (the recommended layout above) so the key never lands on disk.

AI-mode flags #

FlagEffect
--no-cacheBypass the response cache for this run (still saves to cache).
--dry-runEstimate per-failure cost; no AI call is made.
--config <path>Use a non-default pgwen.conf path.
--output <dir>Output directory (default pgwen/output).
--json-out <path>In addition to the Markdown sent to stdout, persist a structured JSON array of {failure, classification, output, instances} entries to <path>. This is the input format consumed by pgwen-fix in suggest-only mode.
--no-pattern-groupingEscape hatch — disable failure-pattern grouping and call the AI once per failure instead of once per locator. Use only when you suspect the grouper is over-merging unrelated failures.
--provider <name>Choose the AI provider for this run. One of claude (default), openai, azure-openai, copilot. Overrides pgwen.diagnose.ai.provider in config. See Providers below for env vars + extras each one needs.

AI providers #

Both pgwen diagnose and pgwen new route through a common adapter layer. Selection precedence: --provider flag (or PGWEN_AI_PROVIDER env for pgwen new), then pgwen.diagnose.ai.provider config, then default claude. Each provider needs its own API-key env var:

ProviderKey env varDefault modelExtras
claude (default)ANTHROPIC_API_KEYclaude-sonnet-4-6—
openaiOPENAI_API_KEYgpt-4o-2024-08-06—
azure-openaiAZURE_OPENAI_API_KEY(deployment-defined)Requires pgwen.diagnose.ai.azureOpenai.deployment + apiVersion + (resource or baseUrl).
copilotGITHUB_TOKENgpt-4oAdapter exchanges the GitHub PAT for a short-lived Copilot bearer (~25 min, auto-refreshed). Copilot's chat API is OpenAI-compatible.

Model override per provider: pgwen.diagnose.ai.openai.model, .azureOpenai.model, .copilot.model, etc. Pricing for each provider's models lives in BudgetGate's DEFAULT_PRICING table — see Settings → AI-assisted diagnose for the full list.

Pattern-level grouping #

Default behaviour in AI mode: failures are grouped by a conservative fingerprint before the AI pipeline runs. Two failures share a group when they share all four of:

  • Rule-based classification class (e.g. LOCATOR_NOT_FOUND);
  • Locator binding file:line (resolved via LocatorIndex);
  • Selector strategy (id, css, etc.);
  • Selector value.

One representative per group goes to Claude on behalf of all members (lex-smallest by feature file / scenario name / step text — stable across reruns so the response cache stays effective). The resulting DiagnoseOutput is fanned out to every group instance.

When the locator binding cannot be resolved for a given failure (no LocatorIndex match), the fingerprint falls back to a per-failure unique key — i.e. no grouping. This is intentional: we never merge across patterns we cannot precisely identify.

Reach for --no-pattern-grouping when you genuinely want one Claude call per failure (e.g. a research run measuring per-instance variance). Costs scale linearly with failure count in that mode.

Suggest-only fixes via @pgwen/fix #

pgwen diagnose never writes patches itself. The companion peer-package @pgwen/fix consumes the --json-out file and, for every Claude proposal that passes its safety gates, writes:

  • a structured suggestion.json capturing the change + its blast radius (affected_instances);
  • a suggestion.patch applyable with patch -p1;
  • a self-contained reports/pgwen-fix/index.html that mirrors the core pgwen look-and-feel.

End-to-end pipeline:

$ yarn pgwen                                                  # produces results.json
$ yarn pgwen diagnose --input r.json --json-out out.json      # Markdown + JSON
$ npx @pgwen/fix --input out.json                             # writes reports/pgwen-fix/

Suggest-only mode never touches source files or branches; the operator reviews each card in the HTML report and applies (or rejects) by hand. The branch / PR auto-apply mode in @pgwen/fix remains a §14 skeleton — gated on the strategy-doc governance answers in pgwen-ai-fix-governance-questions.md.

Annotation workflow #

The classifier's false-positive rate is the gate the strategy doc names for shipping auto-patching (@pgwen/fix) — the threshold is < 2%. To compute it, reviewers tag each historic failure with what the human actually did and whether the classifier was right. pgwen diagnose --annotate walks unannotated entries and prompts for both: 

$ pgwen diagnose --annotate
Scanning pgwen/output/reports/diagnosis-history/
  3 already annotated · 12 pending review

[1/12] Login__Valid-credentials__2026-06-15T10-00-00-000Z.json
  Feature:    Login  (features/Login.feature)
  Scenario:   Valid credentials
  Step:       When I click the submit button
  Error:      locator.click: Timeout 5000ms exceeded
  Classifier: LOCATOR_NOT_FOUND (high)
  Signals:    errorClass=TimeoutError; handlerCategory=locator-action

  Human action? [l]ocator / [a]pp / [n]o action / [o]ther / [s]kip / [q]uit: l
  Outcome? [t]rue positive / [f]alse positive / [a]cceptable / [s]kip / [q]uit: t
  Saved: human_action=fixed_locator outcome=true_positive

Use --limit N to cap the session length. Existing annotations are preserved across runs — only entries without an outcome field are surfaced.

Key mappings:

KeyMaps to
lhuman_action = "fixed_locator"
ahuman_action = "fixed_app"
nhuman_action = "no_action"
ohuman_action = "other"

Telemetry report #

Once entries are accumulating (annotated or not), pgwen diagnose --report produces a Markdown summary that doesn't depend on results.json — it reads only the sidecars under <output>/reports/diagnosis-history/ and <output>/reports/diagnosis-cache/. 

$ pgwen diagnose --report --output pgwen/output

Sidecar file format #

Each failed leaf step gets one JSON file under <output>/reports/diagnosis-history/ (opt-in via pgwen.diagnose.history.enabled = true). Filename format: <feature-slug>__<scenario-slug>__<isoStamp>.json. 

{
  "pgwen_version": "1.0.0",
  "timestamp": "2026-06-15T10:00:00.000Z",
  "feature_file": "features/Login.feature",
  "feature_name": "Login",
  "scenario_name": "Valid credentials",
  "step_keyword": "When",
  "step_text": "I click the submit button",
  "error_class": "TimeoutError",
  "error_message": "locator.click: Timeout 5000ms exceeded",
  "classification": {
    "class": "LOCATOR_NOT_FOUND",
    "confidence": "high",
    "signals": ["errorClass=TimeoutError", "handlerCategory=locator-action"]
  },
  "human_action": "fixed_locator",   // ← filled in by --annotate
  "outcome": "true_positive"         // ← filled in by --annotate
}

External tooling can populate human_action + outcome directly (a git-log heuristic, an annotation web app, etc.) — pgwen's analysis layer accepts whatever wrote the file as long as the schema matches.
