pgwen new — Claude-powered project scaffolder

npx pgwen new is an interactive terminal utility that asks a small set of questions about a new project (capped at 12 total — anything it can't ask about goes into a TODO.md for you to fill in later), calls your chosen AI provider with the answers, proposes a blueprint you must approve, and only then writes the project repository baseline to disk: feature, meta, conf, package.json, README, optional TODO.md, and an initial git commit.

Consumer command: npx pgwen new (or yarn pgwen new, or pnpm pgwen new) — space, not colon. The colon variant pgwen:new only appears inside the pgwen framework's own package.json as a dev shorthand and is not what end users type. See Installation → pgwen command.
It's a starting point, not a finished project. Generated Gherkin and locators will need a manual pass before a real run. Treat the output like a senior author's first draft — useful structure, but always review the meta locators and feature scenarios. Anything the wizard couldn't answer ends up in TODO.md as a checklist.

Prerequisites #

  • Working pgwen install (see Installation)
  • An API key for your chosen AI provider — defaults to Claude (ANTHROPIC_API_KEY). See Providers below for OpenAI / Azure OpenAI / GitHub Copilot.
  • A template project repo to copy fixed files from. The path is configurable via --template <path>; the utility falls back to a sibling directory if not given.

Run it #

export ANTHROPIC_API_KEY="sk-ant-..."
npx pgwen new

# Pick a different AI provider (default is Claude):
export OPENAI_API_KEY="sk-..."
npx pgwen new --provider openai

# Pin the provider via env instead of the flag — convenient for CI:
export PGWEN_AI_PROVIDER=openai
npx pgwen new

# Modes (--mode flag):
#   guided  (default)  step-by-step questions with help text
#   paste              skip the granular questions; paste one requirements blob
#   expert              sensible defaults; only ask project name + paste block
npx pgwen new --mode paste

# Or override the template + output paths:
npx pgwen new --template ../path/to/your-template --output ../

# Inline your organisation's conventions (login flow, results tracking, naming):
npx pgwen new --conventions ./org-conventions.md

# Feed requirements documents (repeatable — .md / .txt / .pdf; file or directory):
npx pgwen new --doc ./requirements.pdf --doc ./docs/specs/

# Feed meeting transcripts / notes (repeatable):
npx pgwen new --transcript ./kickoff-transcript.md

# Mimic the shape of an existing pgwen project (naming, meta layout, profiles):
npx pgwen new --reference-project ../my-earlier-project

# Pull a URL as a reference (HTML → text, or application/pdf):
npx pgwen new --doc-url https://example.com/api-spec

# Bypass the hard context-window guard (~100k-token cap) when your provider handles more:
npx pgwen new --doc ./enormous-spec.pdf --allow-oversized

Reference materials #

Beyond the interactive Q&A, the wizard accepts five kinds of external reference that are injected into Claude's system prompt in dedicated, labelled blocks. Each kind carries a guidance line telling Claude how to weight it:

FlagRepeatableKindHow Claude weights it
--conventions <path>no (file or dir)Design conventionsMUST be honoured — overrides the generic defaults.
--doc <path>yes (file or dir)Requirements documentationExtract explicit requirements, acceptance criteria, edge cases.
--transcript <path>yes (file or dir)Meeting transcripts / notesWeight speaker intent and consensus decisions; ignore off-topic chatter.
--reference-project <path>no (dir)Reference project patternMimic the shape — naming, capability distribution, StepDef style. Do NOT copy scenarios verbatim.
--doc-url <url>yesExternal web referencesTreat as documentation the user has explicitly cited.

Supported file types for --conventions, --doc, --transcript: .md, .markdown, .txt, .pdf, .docx. Directories are walked recursively; node_modules, .git, dist, coverage, .cache, .yarn are skipped. Per-file limit: 5 MB / 500 000 chars post-extraction. Aggregate soft-warns at ~40 k tokens, hard-fails at ~100 k tokens (override with --allow-oversized).

DOCX text is extracted via a built-in ZIP + XML reader — no external dependencies. Password-protected documents are refused with a clear error.

AI providers #

pgwen new (and pgwen diagnose — see Diagnose) talks to AI through a common adapter layer. Pick the provider via --provider <name>, PGWEN_AI_PROVIDER env, or pgwen.diagnose.ai.provider config (precedence in that order; default claude). Each provider needs its own API-key env var:

ProviderKey env varExtras
claude (default)ANTHROPIC_API_KEY—
openaiOPENAI_API_KEY—
azure-openaiAZURE_OPENAI_API_KEYAlso requires AZURE_OPENAI_DEPLOYMENT, AZURE_OPENAI_API_VERSION (default 2024-08-01-preview), and one of AZURE_OPENAI_RESOURCE or AZURE_OPENAI_BASE_URL.
copilotGITHUB_TOKENAdapter exchanges the GitHub PAT for a short-lived Copilot bearer.

Question cap (12) + TODO.md #

pgwen new asks at most 12 questions total across the whole session — enough to anchor the design, few enough that non-technical authors don't get fatigue. Anything the AI couldn't ask about (or chose not to, because answering wouldn't materially change the design) goes into the blueprint's todos array. After you approve the blueprint, those entries are written to TODO.md at the project repo root as a - [ ] checklist: 

# TODO — my-project

This project was scaffolded by `pgwen new`. The items below are things the AI
could NOT infer from the questionnaire and need your manual attention
before the project will work end-to-end.

Tick each box once done. Anything you discard, delete the line.

- [ ] Set pgwen.notifications.slack.webhookUrl in pgwen/conf/env/prod.json to the
      channel you want failure pings in (#your-alerts-channel recommended).
- [ ] Add the SSO secret to .env as PORTAL_SSO_TOKEN. The project's
      login step reads it at runtime.

The cap is enforced client-side. If the AI keeps asking, the CLI prints a ⚠ Question cap reached notice and forces the next turn to emit a blueprint with TODOs for the unknowns. Per-question UI also shows (question N/12) so you can pace expectations.

Conversation flow #

Every run progresses through up to four kinds of Claude turn before generation. The CLI mode (guided / paste / expert) only changes how many questions Claude needs to ask — the blueprint approval step is mandatory in every mode.

  1. Project type (asked first). Claude must determine whether the project is web-ui, api, rpa, or mixed before doing anything else. Other questions branch off this answer. rpa is the typical automation pattern: CSV-in, orchestration steps, results-out, often headless.
  2. Batched questions (up to 6 per turn). Whenever 2+ unrelated answers are missing, Claude bundles them so you answer once per round-trip rather than once per question.
  3. Warnings (proactive). If Claude spots a risk before generating (high-volume run without --parallel; secrets without :masked; etc.), it surfaces a warning with concrete next-step options — including Accept and continue.
  4. Blueprint (mandatory). Before any files are written, Claude emits a structured blueprint covering folder structure, selected capabilities (✓), excluded capabilities (·), npm/yarn scripts, sample-file previews, CI/CD, assumptions, and ⚠ risks. The CLI prompts "Approve this blueprint? (Y/n)". Reject and you can dictate revisions; Claude regenerates the blueprint. Only after explicit approval does Claude emit the ready response.
The blueprint gate is unconditional. If Claude attempts to skip straight to file generation (even in --mode expert), the CLI intercepts the response and asks Claude to emit a blueprint first. The CLI never writes files without an approved blueprint on record for the current session.

Reviewing the blueprint #

The blueprint renders as a structured panel with sections grouped by pgwen blue (#1f23ae) headers. It looks roughly like this: 

  ┌─ Blueprint ────────────────────────────────────────────────────────
  │ A project that searches an account by number and verifies the result
  │
  │ Folder structure:
  │   account-project/
  │     pgwen.conf
  │     package.json
  │     pgwen/
  │       features/Example.feature
  │       meta/Example.meta
  │       conf/env/{test,prod}.json
  │       input/input-feed.csv
  │
  │ Selected capabilities:
  │   ✓ CSV-driven data feed
  │   ✓ @Try wrapping on the search step
  │   ✓ Cron schedule (0 6 * * 1-5)
  │   ✓ Notification webhook (email / chat platform) on test + prod
  │
  │ Excluded capabilities:
  │   · Parallel mode — daily volume below 1000 records
  │   · :masked secrets — no inline credentials in this project
  │
  │ Scripts:
  │   pgwen: pgwen -p Example
  │   pgwen:dryRun: pgwen -p Example -bn
  │   pgwen:repl: pgwen --repl
  │
  │ Sample files:
  │   pgwen/features/Example.feature
  │       Feature: An account is searched - ${RECORD_ID}
  │         Scenario: ...
  │   pgwen/meta/Example.meta
  │       Feature: Example meta
  │         @StepDef Scenario: ...
  │
  │ CI/CD:
  │   Pipeline (Jenkins / GitHub Actions / Azure Pipelines) with daily 06:00 trigger; notify via webhook on failure.
  │
  │ Assumptions:
  │   · Test env URL lives in conf/env/test.json
  │   · Account-number field has a stable data-test-id attribute
  │
  │ Risks:
  │   ⚠ Locator selectors are heuristic — review before first run
  │   ⚠ No retry beyond the @Try wrapper; consider @Retry if upstream flakes
  └────────────────────────────────────────────────────────────────────
  Approve this blueprint? (Y/n)

Answer n and type your revisions — Claude regenerates the blueprint with your feedback applied. Approve to proceed to file generation.

Drive the wizard offline (no API key) #

Before you have an Anthropic key on a machine, you can still walk the wizard end-to-end against the bundled MockAdapter. Useful for demos, CI, or local development: 

PGWEN_AI_PROVIDER=mock \
PGWEN_AI_MOCK_FIXTURES=./fixtures/ai-mock/new-project-interview.json \
yarn pgwen new --project-name example-csv-uploader \
               --description "Upload a CSV to the test portal and verify the row"

The new-project-interview.json fixture covers a canonical 4-turn flow: question → blueprint → ready (with the full file map). Once an API key arrives, set ANTHROPIC_API_KEY=<key> + drop the PGWEN_AI_* overrides and the same CLI runs against live Claude with identical I/O contract (snapshot-tested in tests/unit/cli/NewProjectPrompt.snapshot.test.ts).

A scenario from a real generated project #

pgwen new generates feature + .meta pairs. The .feature stays a thin user-flow narrative; the .meta carries the reusable @StepDef bodies: 

# pgwen/features/upload.feature
@Examples(file='pgwen/input/uploads.csv')
Feature: Upload CSV to portal

  Scenario Outline: upload <filename> and verify
    Given I log in to the portal
     When I upload "<filename>" via the portal
     Then the result row can be located by xpath "//tr/td[contains(text(),'<filename>')]"
      And the result row should be displayed
      And @Finally @Try I clean up "<filename>"
# pgwen/meta/Upload.meta
Feature: Upload meta

  @StepDef @Context
  Scenario: I upload "<file>" via the portal
    Given the file input can be located by css "input[type='file']"
      And the submit button can be located by id "upload-submit"
     When I send file "pgwen/input/$<file>" to the file input
      And I click the submit button

What gets generated #

Output lands at <output-parent>/<project-name>/. The exact layout depends on the template, but a typical structure is:

<project-name>/
├── pgwen.conf                     ← HOCON settings, project-specific
├── package.json                   ← scripts (pgwen, pgwen:dryRun, …), name + description
├── README.md                      ← Claude-generated overview
├── Jenkinsfile / azure-pipelines.yml ← CI templates (whichever the template provides)
└── pgwen/
    ├── features/
    │   └── <BotName>.feature    ← Claude-generated scenarios
    ├── meta/
    │   ├── <BotName>.meta       ← Claude-generated StepDefs + locators
    │   ├── App.meta               ← copied from template (shared app login etc.)
    │   └── Results.meta           ← copied from template
    ├── conf/
    │   ├── env/
    │   │   ├── test.json          ← notification channels, app URLs
    │   │   └── prod.json
    │   ├── browsers/              ← copied from template
    │   └── profiles/              ← per-integration profile files if flagged
    ├── input/.gitkeep
    └── output/                    ← (gitignored)

The new directory is initialised as a git repo with one initial commit so you can branch off immediately.

Naming convention Output uses pgwen-canonical layout (pgwen.conf, pgwen/ subdir, pgwen.* settings). No pgwen migrate step needed before the first run.

Claude API call #

  • Raw HTTPS POST to https://api.anthropic.com/v1/messages — no SDK dependency.
  • Model: claude-sonnet-4-6.
  • The system prompt carries pgwen DSL knowledge, the project-type axis, the batched-questions contract, the mandatory blueprint gate, and the template's project conventions (auth pattern, results pattern, conf structure, output-format contract).
  • The user message contains the structured Q&A context plus the free-text requirements (and any screenshots as base64 multimodal content blocks).
  • Claude returns one of four shapes per turn — question (singular or batched questions[]), warning, blueprint, or ready. The CLI dispatches on type; only a ready response preceded by an approved blueprint causes files to be written.

After generation #

The utility prints "Next steps" pointing you at the files most likely to need a tuning pass:

  1. Review locators in pgwen/meta/<BotName>.meta — Claude's selector guesses are heuristic, not validated.
  2. Update pgwen/conf/env/*.json with real channel names and app URLs.
  3. Add your input data to pgwen/input/input-feed.csv.
  4. Run yarn pgwen:dryRun to validate every step resolves.

Troubleshooting #

SymptomFix
ANTHROPIC_API_KEY environment variable not setExport the env var before running, or add it to your shell rc.
Template directory not foundPass --template <path> pointing at a known good project repo.
Claude returns malformed JSONRe-run; the system prompt enforces JSON-only output but transient API issues occasionally slip through. The Q&A answers are not persisted between runs, so save your requirements to a file if you want to retry without re-typing.
Generated step "Undefined step" on dryRunThe project uses a DSL pattern pgwen doesn't yet implement. Cross-check against the DSL reference. The most common gap is condition vocabulary (is blank / is defined / should be true) — see the callout on the DSL page.
