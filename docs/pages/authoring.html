Authoring features

Best practices for writing maintainable .feature and .meta files. These aren't enforced by the engine — they're conventions that keep projects readable as they grow.

Declarative over imperative #

Prefer features that describe what the project does, not how. Keep the imperative detail in meta StepDefs.

# Declarative — read at a glance
Scenario: A new customer can subscribe
  Given I am on the signup page
  When  I subscribe as "alice@example.com"
  Then  the welcome email should be queued

# Imperative — leaks UI detail into the feature
Scenario: A new customer can subscribe
  Given I navigate to "/signup"
  When  I enter "alice@example.com" in the email input
  And   I click the submit button
  And   I wait for 2 seconds
  Then  the email queue endpoint should return 200

Naming #

  • Feature files: noun-phrase, PascalCase or kebab-case (whichever you pick — be consistent). SignupFlow.feature, signup-flow.feature.
  • Scenario names: present-tense sentences. "A new customer can subscribe", not "Test signup".
  • Locator binding names: lowercase noun phrases. "the email input", not "emailInput".

Meta files #

Put reusable StepDefs in .meta files. A meta file lives either next to its feature (associative — SignupFlow.feature + SignupFlow.meta) or in pgwen/meta/ for shared cross-feature steps.

Build a step vocabulary #

Wrap repeated low-level sequences into a single high-level step. The feature reads at the right altitude:

# In a shared meta file
@StepDef
@Action
Scenario: I subscribe as "<EMAIL>"
  Given I navigate to "/signup"
  When  I enter "<EMAIL>" in the email input
  And   I click the submit button

Use role keywords #

Annotate StepDefs with @Context, @Action, @Assertion. The default pgwen.behaviour.rules = strict then catches accidental misuse like an action called from a Given.

Bind early, refer often #

Set locator and config bindings up-front; later steps stay short.

# Bindings (.meta)
Feature: Signup bindings

  @StepDef
  Scenario: the email input    can be located by id "email"
  @StepDef
  Scenario: the submit button  can be located by css "button[type=submit]"
  @StepDef
  Scenario: the welcome banner can be located by class "welcome"

Data-driven scenarios #

Drive variants with Examples: tables or a -i CSV/JSON feed instead of duplicating scenarios. See Data-driven runs.

Background sparingly #

Background: is great for "set up the page" boilerplate. If a Background runs for every scenario in a feature and contains 5+ steps, consider a single high-level StepDef instead.

Fail fast on environment errors #

If a feature depends on a config key, assert it explicitly so a misconfigured env errors on the first scenario instead of silently misbehaving.

Given app url is defined by config "app.url"
And   app url should not be empty

Run the formatter #

Run yarn pgwen format in CI (or as a pre-commit hook) to keep keyword alignment consistent. It's idempotent.
