Reports

Every run produces four report formats by default, written under ${pgwen.outDir}/reports/. The HTML report streams per-feature, so partial output is visible immediately after each feature completes.

Formats #

FormatOutputPurpose
HTMLreports/html/index.html + per-feature pagesHuman-readable run report with attachments and screenshots.
JUnit XMLreports/junit/TEST-*.xmlFor CI dashboards (Jenkins, Azure Pipelines).
JSONreports/json/results.jsonMachine-readable run summary.
CSVreports/csv/results.csvFlat row-per-scenario summary.

HTML report #

The HTML report includes:

  • An index page summarising every feature with pass/fail status, duration, and a link
  • Per-feature pages with collapsible scenario/step trees
  • Parameters and Attachments dropdowns on the outer step row (see below)
  • Failed-step attachments dropdown (screenshot, error message, page HTML, PDF text capture if relevant)
  • A coloured failure-class badge on each failed step
  • Inline data feed records for data-driven scenarios
  • Step de-duplication when pgwen.web.capture.screenshots.duplicates = false

Failure-class badge #

Every failed step is annotated with a rule-based failure class. The label appears inside the red error panel, formatted as <CLASS> · <confidence>. Hover the badge to see the signals that fired.

ClassBootstrap labelTypical signals
LOCATOR_NOT_FOUNDlabel-warningTimeoutError + locator.click/fill/… in the error
ASSERTION_FAILEDlabel-dangerDslAssertionError OR handlerCategory=assertion
TIMEOUTlabel-infoTimeoutError on a non-locator wait
AUTH_FAILURElabel-warningHTTP 401/403, login-flavoured URL, "invalid credentials" text
NAVIGATION_FAILURElabel-dangerpage.goto + net::ERR_* or HTTP 5xx
UNKNOWNlabel-defaultfallthrough — AI diagnose adds value

The same field is emitted on each failed step in results.json as failureClass: { class, confidence, signals }, ready for pgwen diagnose to consume.

Sibling-success-rate chip #

When the classifier reclassifies an ASSERTION_FAILED step to LOCATOR_NOT_FOUND via its locator-ancestor lookup, it records how often the same binding name succeeded in sibling scenarios. The HTML report surfaces this as a coloured chip next to the failure-class badge:

RateChip colourWhat it tells you
≥ 0.8label-successBinding mostly works — likely an environmental or app-side issue, NOT a selector drift.
0.3 – 0.8label-warningMixed sibling success — manual review recommended.
< 0.3label-dangerBinding broadly broken — probable selector drift.

Hovering the chip reveals the underlying ratio (N/M) and the interpretation hint. Absent when the underlying signal isn't recorded (single-scenario runs, non-locator failures, older runs).

@pgwen/fix suggestions chip #

When a sibling pgwen-fix/suggestions/*.json directory exists next to the report, the index page renders a footer chip: 

┌──────────────────────────────────────────────────────────────┐
│  [pgwen-fix]  3 suggestions available · open pgwen-fix report │
└──────────────────────────────────────────────────────────────┘

The chip links to pgwen-fix/index.html (or ../pgwen-fix/index.html when the main report is in a html/ subdirectory). Chip is omitted when no suggestions exist — the main report stays unchanged.

Parameters and Attachments #

Steps that expand to a StepDef render two dropdowns on the outer collapsible row:

  • Parameters — caller-supplied parameter bindings (e.g. name : Statements for I open the "Statements" tab) and row context for @Examples iterations.
  • Attachments — every scope binding captured during the step's execution, including bindings made by nested StepDef steps. Each link opens the captured value in a new tab.

Attachment file names follow <seq>-<name>-<nameHash>-<contentHash>.txt. The two hashes are 64-bit FNV-1a of the binding name and value, so a binding aggregated up the StepDef tree appears under one physical file referenced from both parent and child dropdowns.

Formatted-text attachments #

Steps that produce a formatted value (typically @DateTime or @Number format steps, or the bare is the current date formatted as "<fmt>" form) surface the formatted output as an Attachments entry on the step that wrote it. The value is captured automatically via the Compositor's scope-write buffer — project authors don't need to do anything extra. Example step + resulting attachment: 

And the raw date is "20260629"
And @DateTime I format the raw date from "yyyyMMdd" to "dd-MM-yyyy" as the iso date

→ Attachments dropdown on the @DateTime step:
    the iso date : "29-06-2026"
    the raw date : "20260629"

Results CSVs #

Custom flat CSVs for project-specific outcome reporting. Configure under pgwen.report.results:

pgwen.report.results {
  fields {
    body = [
      { field = "STATUS", ref = "pgwen.feature.eval.status.keyword.upperCased" }
      { field = "ORDER_ID" }
      # Sort the output by DURATION_MS ascending at end of run.
      { field = "DURATION_MS", ref = "pgwen.feature.eval.duration.msecs", sort = "ascending" }
    ]
    reason = [
      { field = "FAILED_REASON", ref = "pgwen.feature.eval.status.message" }
    ]
  }
  files {
    passed { file = "${pgwen.outDir}/results-PASSED.csv"; scope = "Feature"; status = "Passed"; fields = [ ${pgwen.report.results.fields.body} ] }
    failed { file = "${pgwen.outDir}/results-FAILED.csv"; scope = "Feature"; status = "Failed"; fields = [ ${pgwen.report.results.fields.body} ${pgwen.report.results.fields.reason} ] }
    all    { file = "${pgwen.outDir}/results-ALL.csv";    scope = "Feature";                  fields = [ ${pgwen.report.results.fields.body} ${pgwen.report.results.fields.reason} ] }
  }
}

Sort by column #

Any field can declare sort = "ascending" or sort = "descending". At end-of-run, each output file is re-read and the data rows are sorted by that column (header preserved). Sort semantics:

  • Numeric when EVERY data-row value in the sort column parses as a finite Number — so "10" sorts AFTER "9", not before. Matches the reference framework ResultFile.scala.
  • Lexicographic otherwise (string localeCompare).
  • At most one sort field per output file. Two or more is a construction-time error (the reference framework multipleSortFieldsError).

Named results #

Use @Results('name') to route a scenario's outcome to a specific named file declared under pgwen.report.results.files.<name>. The default (untagged) routes to the unnamed files in pgwen.report.results.files.

Implicit bindings #

Reports can interpolate any scope binding — including the pgwen.* implicit values populated at runtime:

BindingMeaning
pgwen.feature.name / pgwen.scenario.nameCurrent feature/scenario name
pgwen.feature.eval.status.keywordPassed / Failed / Pending
pgwen.feature.eval.status.keyword.upperCasedPASSED / FAILED
pgwen.feature.eval.duration.secsElapsed seconds for the feature
pgwen.feature.eval.status.messageError message on failure
pgwen.feature.eval.sequenceNo1-based feature index within the run
pgwen.now / pgwen.now:<format>Current ISO timestamp; :yyyy-MM-dd etc.
pgwen.downloadDirBrowser download directory for this context

Suppressing meta pages #

pgwen.report.suppress.meta = true (default) hides StepDef-only meta features from the HTML index — useful when shared meta libraries are larger than the feature surface itself.
