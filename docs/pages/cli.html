CLI

pgwen [options] [features...]
pgwen format    [--pretty] [--check] [path...]
pgwen migrate   [--dry-run] [path]
pgwen diagnose  [--rules-only | --report | --annotate] [flags]
pgwen new       (scaffold a new project — see pgwen new)
pgwen-fix       (peer package; consumes diagnose --json-out)

FlagDescription

format sub-command #

pgwen format [--pretty] [--check] [path...]
  • --pretty (default) — rewrite .feature/.meta files in place with normalised keyword alignment.
  • --check — exit non-zero if any file would change. Use in CI.
  • path — files or directories (default: pgwen/).

migrate sub-command #

pgwen migrate [--dry-run] [path]

One-shot, idempotent rename of an existing project directory to the canonical pgwen layout:

  • Renames any legacy pgwen.conf to pgwen.conf and the pgwen/ subdir to pgwen/.
  • Rewrites HOCON block names, key prefixes, and ${...} substitutions.
  • Rewrites .feature / .meta interpolations.
  • Rewrites package.json scripts and CI files (Jenkinsfile, azure-pipelines.yml).
  • --dry-run previews changes without writing.

diagnose sub-command #

pgwen diagnose [--rules-only | --report | --annotate]
               [--input <results.json>] [--output <dir>] [--config <pgwen.conf>]
               [--no-cache] [--dry-run] [--limit <N>]
               [--json-out <path>] [--no-pattern-grouping]

Reports on failures classified by the rule-based engine; optionally invokes Claude for AI-assisted diagnosis when pgwen.diagnose.ai.enabled = true and a key is resolved. Falls back to --rules-only with a notice when any gate fails — see the full guide at Diagnose & AI → pgwen diagnose CLI.

  • --rules-only — offline, rule-based per-failure report.
  • --report — aggregate telemetry analysis from diagnosis-history/ + diagnosis-cache/.
  • --annotate — interactive review of history entries (human_action + outcome).
  • --no-cache — bypass the response cache for AI mode.
  • --dry-run — estimate per-failure cost; no Claude call.
  • --limit N — cap the number of entries during --annotate.
  • --json-out <path> — persist a structured FixInputEntry[] JSON file (the input format consumed by pgwen-fix) alongside the Markdown stdout.
  • --no-pattern-grouping — escape hatch; disable failure-pattern grouping in AI mode so the AI is called once per failure rather than once per locator pattern.
  • --provider <name> — pick the AI provider for this run: claude (default) / openai / azure-openai / copilot. Overrides pgwen.diagnose.ai.provider in config. See Diagnose → Providers for env vars + extras each one needs.

pgwen-fix sub-command #

# Suggest-only mode (the default) — writes .json + .patch sidecars.
pgwen-fix --input <diagnose-output.json>
          [--reports-dir <path>]
          [--confidence-minimum high|medium]
          [--diff-max-lines <N>]
          [--dry-run]

# Auto-apply mode — rewrites the line, creates a pgwen-fix/<slug>-<utc>
# branch, commits. Local only; no push, no PR.
pgwen-fix --input <diagnose-output.json>
          --auto-apply
          [--allowed-path-prefix <p>]   (repeatable; default: pgwen/meta/)
          [--diff-max-lines <N>]
          [--dry-run]

The companion peer-package CLI (ships in @pgwen/fix, structurally separate from core pgwen). Consumes the JSON file produced by pgwen diagnose --json-out.

Suggest-only mode (default) writes:

  • One .json + one .patch per accepted suggestion in <reports-dir>/suggestions/;
  • A self-contained <reports-dir>/index.html matching the core pgwen visual style;
  • An append-only <reports-dir>/history.jsonl ledger backing the repeat-fix circuit breaker.

Never modifies source files or branches. Optionally posts (or idempotently updates) a single GitHub PR comment when GITHUB_TOKEN, PGWEN_FIX_GH_REPO, and PGWEN_FIX_GH_PR are all set in the environment.

--auto-apply mode runs the per-entry pipeline validateMinimumDiff → detectPriorPgwenFix → applyMachineProposal:

  • Validates the proposal against the configured path allow-list (default pgwen/meta/) + minimum-diff + binding-name checks;
  • Refuses if a prior pgwen-fix: commit already touched the same line within the look-back window (circuit breaker);
  • Rewrites the line, creates a pgwen-fix/<slug>-<utc> branch, commits. Never pushes; never opens a PR — operator inspects + pushes manually.
  • Combine with --dry-run for a no-side-effect rehearsal ("would apply" output).

Per-entry rejections don't stop the batch; the exit code reflects whether any entry was rejected (0 = all applied, 1 = ≥1 rejected). See Settings → @pgwen/fix for every config key and the full pipeline gates.

Examples #

# Run the smoke-tagged scenarios from the todo profile in headed mode
yarn pgwen -p todo --tags @smoke --headed

# Parallel run with a CSV feed
yarn pgwen --parallel -i pgwen/input/customers.csv pgwen/features/

# Dry-run all features in CI
yarn pgwen -bn

# Format every .feature / .meta file in place
yarn pgwen format

# Migrate a project dir (preview first)
yarn pgwen migrate --dry-run ./my-project
yarn pgwen migrate ./my-project

# Rule-based diagnose on the last run
yarn pgwen diagnose --rules-only

# AI-assisted diagnose (requires pgwen.diagnose.ai.enabled = true)
ANTHROPIC_API_KEY=sk-ant-... yarn pgwen diagnose

# Estimate AI cost without calling Claude
yarn pgwen diagnose --dry-run

# Walk historic failures, capture human_action + outcome
yarn pgwen diagnose --annotate --limit 5

# Print the aggregate telemetry report (false-positive rate, model usage, ...)
yarn pgwen diagnose --report

# Persist machine-proposal JSON for pgwen-fix to consume
yarn pgwen diagnose --json-out pgwen/output/diagnose.json

# Generate suggestion sidecars + HTML report from a diagnose --json-out file
npx pgwen-fix --input pgwen/output/diagnose.json

# Auto-apply mode — creates a pgwen-fix/<slug>-<utc> branch + commit per
# safe locator-drift entry. Inspect + push manually.
npx pgwen-fix --input pgwen/output/diagnose.json --auto-apply

# Auto-apply dry-run — see what would land without writing or branching.
npx pgwen-fix --input pgwen/output/diagnose.json --auto-apply --dry-run

# Auto-apply with a custom path prefix (when project meta lives elsewhere).
npx pgwen-fix --input pgwen/output/diagnose.json --auto-apply \
              --allowed-path-prefix projects/checkout/meta/

# Drive any AI flow through the offline MockAdapter — no Anthropic key,
# no network. Useful for CI, demos, or local development before a key
# is provisioned.
PGWEN_AI_PROVIDER=mock \
PGWEN_AI_MOCK_FIXTURES=./fixtures/ai-mock/all.json \
yarn pgwen diagnose --input pgwen/output/results.json

Mock AI adapter for offline development #

pgwen ships a MockAdapter alongside the four real provider adapters (Claude / OpenAI / Azure OpenAI / GitHub Copilot). Opt in by setting --provider mock or PGWEN_AI_PROVIDER=mock. Mock is not in the production AiProvider enum — it's a deliberate escape hatch that fires only when explicitly requested, so production code paths stay untouched.

Drive it from a JSON fixture file: 

# Combined router fixture — covers every Claude DiagnoseCategory
PGWEN_AI_PROVIDER=mock \
PGWEN_AI_MOCK_FIXTURES=./fixtures/ai-mock/all.json \
yarn pgwen diagnose --input pgwen/output/results.json

Eight golden fixtures live at fixtures/ai-mock/ covering locator_drift, app_regression, timing, env, test_bug, unknown, the multi-turn pgwen new interview, and a combined router. Each entry can impersonate any real provider via response.provider so downstream provider-branching code stays exercised. See fixtures/ai-mock/README.md for the schema.
